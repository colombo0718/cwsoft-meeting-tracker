## 3) 討論摘要

### 主題 1｜011 公司【待確認】— 盤點系統手機版
- **現況**：客戶希望員工未登入狀態也能使用手機盤點，不限使用人員
- **核心討論**：目前架構需登入才能盤點；客戶認為只為盤點購買手機不合理
- **結論**：列入 Q2

### 主題 2｜員工違規操作查詢
- **現況**：洪記客戶老闆娘定期查核員工是否違規修改資料
- **核心討論**：AI 比對銷貨單修改前後差異，只列出有差異的欄位
- **結論**：列入 Q2，需設計報表 UI

### 主題 3｜首例【待確認】— 單一會員消費毛利
- **現況**：客戶要求查詢單一會員毛利，目前只有全體會員關係效益
- **核心討論**：在消費紀錄銷售與維修單後各加一欄「毛利」，底部加總；繳費不計入
- **結論**：羿宏依此方案實作

### 主題 4｜金馬講 — 動態專案型號關鍵字搜尋
- **現況**：型號只有場合下拉篩選，冷門型號找不到
- **核心討論**：改為可輸入關鍵字搜尋；分析 90% 常用下拉即可選到，10% 冷門才需搜尋；加搜尋後可減少重複建商品
- **結論**：羿宏實作，正面改善

### 主題 5｜金馬講 — 特案類商品 Enter 鍵跳轉
- **現況**：按 Enter 直接跳行，但客戶需先跳到「轉換」欄
- **結論**：難度不高，羿宏本週處理

### 主題 6｜優惠券系統 — 壓力測試與生日券規則
- **現況**：羿宏已完成程式碼與資料庫建立
- **核心討論**：
  - **序號預建 vs Policy Base**：序號預建查詢快核銷方便，但資料量大時負擔重；Policy Base 反之
  - 決定先跑 20 萬筆壓力測試，不通過再改非同步 IO
  - 過期後清除資料，可保留短暫查詢期
  - **生日券規則（馬尼）**：每月 15 號發下個月生日券，有效期當月 1 日至月底
  - 彥偉禮拜四拜訪馬尼，羿宏需**禮拜三中午前**提供優惠券初版（畫面或文字說明均可）
  - **優惠券圖示**：士豪用 AI 生成，轉檔為 LINE 圖文選單尺寸；參考馬尼現有紙本/電子券
- **未定**：生日券以外券種規則，禮拜四現場確認

### 主題 7｜AI 工具評估：Claude vs GPT
- **現況**：GPT 深度思考模式速度慢，彥偉考慮切換 Claude
- **核心討論**：
  - Claude 程式正確度較高，速度仍觀察中
  - 小葳底層架構已支援多供應商切換
  - Claude memory 匯出轉移流程：匯出記憶 → 取得轉移 prompt → 貼入新 AI；切換前需通知彥偉匯出
  - 羿宏 GPT 專案短期不急切換
- **結論**：暫不立即切換，待彥偉評估

### 主題 8｜士豪報告：報價單 AI 自動生成
- **成果**：
  - 行動裝置資料已匯入資料庫
  - Claude Code 直接讀取資料庫，整合生成完整 HTML 報表，省去複製貼上操作
  - 介紹 Claude Code 介面與 VS Code / Cursor 差異

### 主題 9｜維護合約管理系統與 AI 比對報表
- **現況**：現有帳務系統缺乏完整維護合約欄位，AI 比對報表剛完成初版
- **核心討論**：
  - 士豪用 Claude 比對公司合約清單 vs 系統資料，找出差異
  - 比對結果：43 家中大部分正確，特殊案例：
    - 洪記：含稅計費，與其他客戶不同，已標記
    - 學甲中山（鋼鐵人體系）：線上付款停用，系統有記錄
    - 部分客戶有開新店補點、或到期月數有小出入
  - 維護合約計費邏輯：雲端月租費（分店數階梯）+ 維護費（每間 250 元/月，年繳）
  - 特殊案例：同心體系（1 老闆多間店合併收費有折扣）
  - 士豪傾向在分店設定檔加欄位；羿宏建議另建資料表；以分店為最細收費單位
  - 維護合約若成功整合，月收入有機會達 300 萬，彥偉表示是公司未來傳承重要資產
- **結論**：士豪先隔離維護合約用另外表單生成，確保數字正確後再整合

### 主題 10｜宇新 / 馬尼 — 維修單報表需求
- **現況**：不只宇新，馬尼等多個客戶都在詢問維修單獨立報表
- **核心討論**：在毛利資料二加入維修單篩選，初步版本不需花太久時間；彥偉希望未來做成維修分析表（接了幾單、換了什麼零件、AI 建議備料）
- **結論**：羿宏先做簡易版（單據篩選 + 統計）；宇新下禮拜再去拜訪確認需求

### 主題 11｜小幫手功能擴充
- **現況**：彥偉使用小幫手查詢桃園新竹客戶時，因電話區碼格式（03、030）導致結果不準確
- **核心討論**：
  - 需改善查詢邏輯，用字串包含方式辨識區碼
  - **名稱辨識問題**：靈異通訊 ≠ 零壹通訊行，但名稱相近容易混淆；設計信心度機制，低於閾值時主動詢問確認，累積確認後結案
  - **合約狀態查詢**：彥偉希望小幫手可即時查詢某客戶是否有繳費、是否有買行動裝置，不再依賴會計查電腦
- **待辦**：士豪處理區碼查詢 bug 及名稱信心度機制

### 主題 12｜小葳 AI 客服架構設計進度
- **現況**：士豪整理完 AI 客服知識庫（34 章、每章 3-4 節）
- **核心討論**：
  - **分流架構**：
    - 基本問題 → 讀知識庫文件 → 直接自動回答
    - 進階問題 → 影子模式（存入資料庫，人工確認後再回答）
    - 廢話/閒聊 → 隨機回應
  - **信心度機制**：初期全部影子模式，累積訓練後，信心度 ≥95% 才允許自動回答
  - **年底目標**：能回答 10% 的問題（先從簡單的「功能在哪裡」、「怎麼下載」開始）
  - 彥偉希望讓 Claude Code 晚上自動巡檢 POS 畫面截圖，幫助建立說明文件
  - 羿宏希望有知識庫瀏覽介面，方便查看目前內容
  - 未來把資料庫結構也餵給 AI，讓它能回答更複雜的操作問題
- **結論**：士豪繼續完善知識庫；影子模式先跑；巡檢截圖士豪已開始進行

### 主題 13｜Claude Code 巡檢模式

- **結論**：持續進行，作為小葳知識庫的資料來源
