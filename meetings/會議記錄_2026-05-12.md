## 3) 討論摘要

### 主題1｜遠端認證機制
- **背景：** 影子模式網頁目前只能在公司內部存取；彥偉希望手機與家裡電腦也能用
- **方案參考：** 遠傳業務員地（後台）— **鎖 IP + 鎖憑證**，換 OS/瀏覽器都要更新
- **設計重點：**
  - 手機憑證安裝（搜尋手機憑證機制）
  - 家裡 IP 固定（與 SSL 同一 IP）
  - 認證期限半年，到期重新認證
  - 訊息走 HTTPS（SSL 已加密），加上身份認證
- **目的：** 老闆不裝 LINE OA 也能看訊息（避免「讀掉了客戶就漏」）
- **彥偉今天目標：** 4 點前若能做出來，就不必再到公司

### 主題2｜影子模式包成手機 app
- 用 WebView 把影子模式網頁包成 iOS/Android 執行檔
- 經過認證的人可以登入、查所有來訊
- 「現在把它鎖在裡面 → 外部存取需求 → 用認證方式擴展」

### 主題3｜服務全景與架構圖
- **痛點：** 老闆第一次看到這麼多服務名稱（11 個），看不懂
- **目前服務組成：**
  - **接管服務（5/7 完成）：** 統一管理 11 個服務，全綁定到 NSS（之前架在 VS Code terminal 不穩）
  - **會計系統**
  - **Gate**（小幫手與資料庫的橋接 API）
  - **OP（會員機制認證）+ 會員綁定**（兩個是同一流程，需重新命名為「智慧門市會員」以區別其他用法）
  - **阿全 / 小幫手**（POS 客服用）
  - **GTP 核心**（任務型對話機器人，影子模式 + 小幫手都以此為核心，僅 config/任務數不同）
  - **影子模式（前端 UI + 後端知識庫 API）**
  - **APA reverse proxy**（之前用 Cloudflare，正在把所有東西遷到自家 APA）
- **彥偉要求：** 畫**樹狀架構圖**讓老闆看得懂；用 AI 整理現有架構成 MD 進知識庫
- **備援機制：** 每台主機都假定可能掛 → 4 個備份點（含士豪自家電腦），但目前測試不夠
- **Google sync：** 對外走 Google，sync 到多個內部點

### 主題4｜Codex vs GTP 對比（核心爭論）
- **觸發點：** 「他在做飛機走之前」（羿宏？）把 Codex 串接後可判斷 70-80% 問題
- **彥偉質疑：** 同樣看 code、看資料庫，Codex 八九不離十，士豪的 GTP「輸一截而且還蠻大一截」
- **士豪解釋兩者職責不同：**
  - **Codex**：互動式（CLI 直入）/ 入市式；每次一個指令是新對話，需 context engineering 注入歷史
  - **GTP**：任務型 → 拆解語意 → 知道要做什麼任務 → 抓對應文件章節餵給 LLM → 產生回覆
  - 「Codex 是看 code 都看完的，從介面當然都可以回；可是要擋到手機/應用，就不一樣」
- **整合方向：**
  - 兩個並行：GTP 做意圖分類 → 把問題路由到對應知識章節 → Codex/LLM 回答
  - Codex 當「糾察隊」做 checkpoint 驗證（給一份 checklist，讓 Codex 從各面向檢查）
  - 最後再加一個 audit agent 審核回覆對不對
- **目前可用 LLM：** Grok + HuggingFace（70B 雲端、速度快）；本地端 Codex/GI

### 主題5｜信心度機制與客服量自動化目標
- **彥偉年度目標（用 AI 取代客服量）：**
  - **2026 年底：10%** 客服量自動移除（基本問題如「怎麼安裝」「帳號密碼」）
  - **2027 年：50–80%**
- **信心度設計：**
  - 信心度 ≥95-98% → 直接自動回（不必人工）
  - 信心度低 → 走影子模式（人工確認）
  - 判斷信心度可由 1 個 / 2 個 / 3 個 AI 投票
- **訓練資料：**
  - 所有對話、會議記錄、客戶問答都是 training data
  - 越早收集、訓練得越早
- **CWSoft 優勢：** 知識庫完整度 > Google 通用型；我們有客戶資料庫、有 POS 全套程式碼

### 主題6｜單一介面 + 切換模式
- **彥偉立場：** 不要做多個 LINE OA、不分前後台
  - 「為什麼一定要分前台後台？只要知道你是誰就好」
  - 第一性原理：權限決定看什麼，不是入口
- **方案：** 單一小幫手介面，用**語音/文字切換模式**
  - 模式範例：影子模式、會計、查工作進度、POS 操作
  - 像 Google 一樣不預設使用者要做什麼，等指令進來再分配
  - 進去顯示 1234 主要模式 → 使用者選 / 或語音說
- **不需 Google 等級資源 → 簡單分類即可**
- **彥偉自評 PoS：** 以前 POS 把前後台分開「現在統合在一起」是進步

### 主題7｜語音辨識客製化 preference
- **問題：** 雅婷把「飛龍加六點」斷成「飛龍家三」（飛龍是客戶名）
- **解決：**
  - 用客戶名稱字典做 mapping correction
  - 語音轉文字時，把已知客戶名（前進、宇新、飛龍等）標記為實體名而非動詞
  - 不仰賴 LINE 內建語音轉文字，自己接「語音訊息進來 → 內部轉換」
- **常見模式學習：** 一次正確之後，同一客戶下次問同類問題（如「我要查所有未收款」）就保持

### 主題8｜工作日誌整合（Matrix Manager）
- 士豪用 **Matrix Manager** 記錄日常 AI 互動 / 工作流水
- **彥偉要求格式：**
  - **週報級簡介**：每天改了什麼 / 改了什麼組件 / 簡單 2 句話
  - 老闆不看細節（修改過程、來回 debug、token 浪費）
  - 想看細節時再問 AI 「列出細節，不要超過 100 字」
- **整合：**
  - 士豪的 GTP 系統 + 羿宏的 Codex 開發都要產出 MD 工作記錄
  - 用 AI 對 AI：偵測兩人有沒有做重複的事、可否抽出共用 component（如認證模塊）
  - 維護一份共用 component（不要各做一份各寫一份）

### 主題9｜POS 內建客服按鈕 + 暗號 ID
- **痛點：** 客戶截圖截不到重點、回報「版本最新」實際上不是
- **方案：** POS 內建「我有問題」按鈕
  - 按下去自動打包：畫面、版本、電腦號碼、IP、暗號 ID
  - 暗號 ID（如 A123）對應該客戶 → 自動查資料庫知道他有幾家分店
- **回覆走 LINE OA**（讓客戶覺得是真人服務）
- **防濫用：** 1 分鐘按 10 次 → 自動降級

### 主題10｜多 source 訊息接收（原「開後門」）
- **原稱「開後門」**：可從程式自動灌訊息進 LINE OA 程式做測試
- **彥偉建議改名：** Multi-source / Manager — 認證過的訊息從任何 source 進來都可
- 不只給工程師壓力測試用，也是 POS 客服按鈕的接收入口

### 主題11｜命名建議
- **GTP** 全名是「general case ...?」太工程
- 應改成更易懂：
  - 「**知識庫回應系統**」（士豪建議）
  - 或「**任務導向對話機器人系統**」（更廣義）
- 任務範圍：客服回答、會計帳戶、工作內容、會議記錄 都可以是其中模式

### 主題12｜業務面與客戶
- **客戶昨天一次加 3 分店** → 已做成 SP create，未來標準作業
- **客戶刪掉問題**：之前有中文亂碼問題（家客戶刪除時）
- **新威會員昨天 12 點訊息**沒回 → 彥偉提醒下午要回（雖然不是上班時間，但不要漏）
- **300 萬目標 vs 150 萬實際：** 彥偉初估 150 萬（保守）；維護合約沒給優惠，舊客戶月租化要靠新功能升級
- **AI 普及 → 基礎軟體售價會降：** 越早把新功能做出來、盡早出去推
- **會員復健價：** 抓「700 塊就好」（彥偉吐槽 700 塊能幹嘛？）

### 主題13｜士豪論文與工作節奏
- 6 月初口試
