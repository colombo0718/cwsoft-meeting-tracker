### 主題1｜小幫手強化（士豪）

- **核心討論：**
  - **獨立執行**：包裝成可跳出 VSCode 直接執行（不要靠 VSCode 驅動）
  - **守護/觀察程式**：crash 前能用 try/except 主動傳訊息通知；統一管理所有任務（小幫手、會員登入等）
  - **新功能 1 — 下架分店**：之前在義風那邊做過，現在做進小幫手
    - 可考慮與「Create Customer」(SP) 合併或共用，但實作上仍是兩個 SP（新增客戶 / 加減分店）
  - **新功能 2 — 修改單一客戶結束時間**：針對信用客戶延期使用
    - 例 1：希爾說 5/1 才能匯款，要把分店到期日延後到 5/1 或 5/2 的 23:59:59
    - 例 2：振興要求延到 5/31 23:59:59
    - 設定一律設在最晚（晚上 23:59:59），不設中間
    - 加點功能已存在，再加上「改日期」功能就基本不用開電腦
    - 月底是最常用時段，**千萬不要月底罷工**
  - **使用說明**：要文件化哪些 .py 程式要先開（彥偉自己進去看不知從哪開）
  - **測試/正式環境隔離**：類似影子模式，測試在自己機器跑，正式才掛上 Cloudflare
  - **偵測機制**：改成每小時問一次即可（避免之前裕航那種類 DOS 攻擊式的高頻偵測）
  - **雙層 LLM 防禦**：**Groker** + **Hugging Face**，其中一家若沒回應跳到另一家；另外 **LMF** 也是免費供應商
    - 兩家都提供 70B 模型，做語意分析夠用（不需要太聰明）
    - 用量不大，先用免費的就好，不必特別買 API
  - **提示工程**（論文後再做）：當使用者問題不明確時，AI 應 repeat 問題並告知缺哪些參數，分次追問，不要硬給單一答案

### 主題3｜優惠券系統 — 最低消費門檻是否含專案傭金

- **問題：** 「兌換與使用限制 → 最低消費金額」設定，要不要把專案的傭金也計入？
- **情境：** 客人辦遠傳 1399 專案（單據金額是 0 元），但傭金可能足以兌換一張 100 元折價券
- **討論結論：**
  - **點數記點門檻**：「**有上專案就不考慮最低消費金額**」（即只要有上專案，傭金就直接給點）
  - **折價券（消費抵用）**：仍按原規則，**不考慮專案**
    - 理由：折價券是用點數換的（≈ 退錢），如果含專案計算，會變成「客人 0 元也能折 100」，等於倒貼
    - 例：滿 2000 折 200，若含專案 → 已經 0 元還要退 200，不合理
- **產出：**
  - 設定頁 checkbox 文案改寫，**不能對 end user 寫「傭金」字眼**
  - 建議寫法：「**有專案 OR 低消金額 X 元**」，門市店員為第一道把關
  - 還要再跟客戶（林一/林漾）確認最終文字
  - 折價券設定畫面**現有的什麼都不用改**

### 主題4｜進階行銷 — 退貨退點規則

- **核心討論：**
  - **單據商品退貨**（通案）：依原銷售單號針對該品項退對應點數（如配件 50 點 → 退 50 點，可部分退）
  - **專案退貨**（特案）：
    - 屬「特許」非通案，一年發生數件
    - 必須**回到原銷售單去手動調整**，不在退貨畫面退
    - 涉及遠傳補貼/保證金/核銷，需遠傳總公司審核（有些有 7 天審閱期）
    - 退貨畫面**只退單據商品**，不處理專案
  - **點數調整**：透過既有的「點數調整」入口（有操作者、原因、理由），**不要在退貨畫面開放退點按鈕**
- **產出：**
  - 退貨流程說明掛在點數設定頁，提醒客戶（林一/林漾）目前作法
  - 點數設定本身不用改

### 主題6｜進階行銷 — 零壹通訊行（靈異）試用流程與 6/1 上線

- **背景：** 零壹（內部代稱「靈異」）尚未匯款，但已答應提供試用，目標 6/1 完整上線
- **「換時間」業務手法：** 表面是給客戶試用，實際是延後正式上線時間，讓系統更穩定再收費
- **試用流程設計：**
  1. **先開設定畫面** → 客戶把會員 36 號匯入
  2. 設定官方帳號 admin 權限
  3. 列出客戶需求清單，撰寫使用說明
  4. **其他功能先不開**（避免 bug 嚇跑客戶）— 給「我們還在匯入過程中」當理由
  5. 內部測試完，把操作影片集中放雲端供客戶觀看
  6. 影片給客戶確認沒問題，請客戶提意見
  7. 16 號（士豪回來後）一併把意見測進去
  8. 都符合客戶預期 → 通知客戶 6/1 匯款，正式上線
- **點數補算：** 試用期累積的銷售單（從 **3/16 起算**），正式啟用時補算回去；什麼時間點匯都可以補
- **穩定性要求：** 不能釋出太誇張的 bug；客戶已沒有 ALL CARD（不易跑掉），但若覺得穩定性太差仍可能回頭
