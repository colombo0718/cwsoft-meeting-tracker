## 3) 討論摘要

### 主題1｜Q2 檢討
- **卡住的兩項**：
  - **公版商品下載**：可下載但還在調整，沒人在用 → 主因「沒有資料源 + 沒推廣」
  - **外季/會計帳戶系統**：基本沒開始，僅做了「自動生成清管單」報表
- 結論：複雜度比預期高，速度太慢；不究責，調整目標

### 主題2｜Crystal Report 拆分
- **痛點：** 每月要把 Crystal Report 關掉 10 分鐘（含 print 整理），那 10 分鐘要列印的人就 GG
- **架構：** 是 web service，有資料夾叫做 `print` 程式碼很複雜，繼承一堆 interface（前任工程師「軟工至上」）
- **流程：** DataSet 壓縮 → POST 上去 → 指定哪個 Report → PDF 下來
- **計畫：** 把 Crystal Report 從 Payment 拆出來獨立模組，丟 ChatGPT 重構（簡化掉 iPrintable 那些不必要 interface）
- **目的：** Payment 重做時不用再考慮 Crystal Report

### 主題3｜Q3 目標（7、8、9 月）
1. 補完 Q2 沒完成的兩項（會計帳戶系統 + 公版商品）
2. **Payment 系統重做**：類似重新蓋一個網站
   - 既有功能砍剩月租為主（年繳、月租、1/3/6 月一次都要保留）
   - 之前 bug：開兩個視窗會掛、2600 變 30 元 → 直接停用會計帳戶
   - 接綠界（綠界開發環境是共用的，不放敏感資料）
3. **Leaf 學新東西**：拿 2–3 週學 Leaf（手機端 ID 連結 / EDM 網頁）

### 主題4｜公版商品推廣
- 彥偉：核心問題不是好不好用，而是「**怎麼讓客戶知道**」
- 公版商品在原版上是可以用的，只是沒人知道、沒資料源
- 爬蟲搞定後可以開始推
- **推送方式：** 後端讀資料庫 table → 用 API 發給各客戶老闆
- 客戶要先「訓媒合」才能用公版下載

### 主題5｜LINE 受眾清單與賴受眾系統
- 已建 `賴受眾清單` 資料庫（在 Share 庫）
- **流程：** 填寫客戶名稱 + 受眾名稱 → 後端讀取 → 自動建立 LINE 受眾
- API 動態建立：填了就直接同步，不需手動轉換
- 之前提到的「e-Sale 匯入」可改為直接填資料庫

### 主題6｜Leaf 學習與「搖錢樹」EDM 網頁
- **彥偉的長遠構想：** 創造「搖錢樹」— 先給客戶 demo 級的免費網頁 → 進來後想加什麼都付錢
- **第一版 demo：** 一個按鈕的 LINE 圖文 → 點開秀現在熱門商品 EDM（電子海報）
  - 不做購物車、不做 login（從 UID 知道是誰）
  - 純 EDM 形式，店員自行上傳圖片
  - 商品圖、WebPrice（網路售價）等都已經有欄位
  - 各客戶風格不同（Style 1/2/3 + 換 banner）
- **賣點：**
  - 門市可以「我知道你是誰、你買過什麼」做客製化推薦
  - LINE OA 連結 + 歷史消費紀錄
- **價格策略：** 免費進場 → 加功能付錢；月加 2,000 元也是穩定收入
- **重要原則：** 大部分客戶要的不是新功能，而是「客人」（流量）
- 彥偉：第一年公司只賺 6 萬要四個人分；要拼

### 主題7｜發票抬頭圖片儲存位置
- 目前圖檔是放在每台 POS 本機，需手動 copy 到每台店面電腦
- **討論結論：** 放在**分店設定**較合理（每店一張圖，不放雜項[全公司共用]、也不放單機[每台一份]）
- **儲存：** 不放資料庫；用既有 File12 / NAS / Google Drive 同步
  - File12 程式碼以前寫好但一直沒啟用
  - 可限制大小 200K
  - 黑白限制（彩色發票機印不出來）

### 主題8｜小幫手（順帶）
- 那天小幫手掛掉是因為內部錯誤（疑似 LINE 網址掉了），重測一次 OK
- 「這個未來是你要處理的」（彥偉對某人說，可能是要交接給士豪）

### 主題9｜彥偉個人/業務策略雜談
- 老闆要付房子錢，需要拼
- 公司要走「搖錢樹」收入模式，不能只靠 2,600 元月費
- 「東西要做得有趣，自然會帶錢來，錢是附加的」
