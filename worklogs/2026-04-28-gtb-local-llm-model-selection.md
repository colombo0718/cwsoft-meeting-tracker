## 討論摘要

### 1. GTB 的本地模型需求不是「最大」，而是「穩定主路徑」

**結論：** GTB 的本地 LLM 核心應優先服務分類、欄位抽取、信心值、JSON 格式輸出、簡短推理與任務路由，不追求大模型聊天感。

補充說明：
- GTB 更像「任務路由器 + 表單抽取器 + 小型參謀」，不是純聊天機器人。
- 因此模型選型重點是：中文能力、格式穩定度、低延遲、可 benchmark、可長時間運行。
- 若要做高風險任務複核，可另外補一顆慢速審查模型，而不是把主路徑全部交給大型 reasoning 模型。

### 2. 個人主機（RTX 2060 6GB / RAM 16GB）適合作為低成本遠端 LLM 供應節點

**結論：** 這台個人主機可以穩定扮演遠端 LLM 供應節點，但定位應放在 3B～8B 級量化模型，不適合直接承擔 14B、20B 以上模型的主力並發。

主機條件：
- CPU：Intel i5-12400F
- RAM：16GB
- GPU：NVIDIA RTX 2060 6GB

補充說明：
- 這台機器的瓶頸是 6GB VRAM，而不是硬碟空間。
- 若模型檔案接近或超過 6GB，通常需要 CPU/RAM offload，延遲會明顯上升。
- 因此它的價值不在「跑最大模型」，而在於變成一台可常駐、低成本、私有化的 LLM worker。

### 3. 本輪優先推薦的本地模型組合

**結論：** GTB / remote_worker 的第一組雙核心，優先測 `qwen3:4b` 與 `gemma3n:e2b`；若要提高品質，再逐步往 `qwen3:8b` 或其他中型模型測。

建議分工：
- `qwen3:4b`
  - 中文分類、欄位抽取、JSON 輸出、GTB 主路徑候選
- `gemma3n:e2b`
  - 輕量對話、摘要、備援回覆、低成本常駐
- `gemma3:4b`
  - 長文整理、一般文字回覆備援
- `phi4-mini`
  - 小型邏輯判斷、格式檢查、信心值校驗
- `qwen3:8b`
  - 可作為品質上限測試，但在 2060 6GB 上不一定適合高頻主路徑
- `deepseek-r1:8b`
  - 適合慢速二次審查，不適合 webhook 主路徑

### 4. Gemma 3n E2B 與個人主機的匹配度高

**結論：** `gemma3n:e2b` 非常適合這台 2060 6GB 主機，值得直接拉下來做 benchmark。

補充說明：
- 它是為低資源裝置場景設計的輕量模型，作為常駐 worker 很合理。
- 可與 `qwen3:4b` 並列測試，分別比較中文抽取準確率、格式穩定度、平均延遲與 timeout 率。

### 5. Hugging Face 的 Gemma 4 E2B 可以測，但不該直接當這台機器的正式主力

**結論：** `google/gemma-4-E2B` 可以列入實驗名單，但不適合直接用 Hugging Face 原始版作為這台 2060 6GB 機器的穩定遠端 LLM 主力。

補充說明：
- 這次釐清：`gemma3n:e2b` 與 Hugging Face 上的 `google/gemma-4-E2B` 不是同一顆模型。
- `google/gemma-4-E2B` 雖是 E2B 等級，但原始權重檔仍偏大，直接跑 Transformers / safetensors 會對 6GB VRAM 造成壓力。
- 若沒有 GGUF / 4-bit 量化版，實務上通常會落到 RAM offload，速度與穩定性都不適合當正式供應核心。
- 若未來出現合適量化版，才適合重新評估是否納入個人主機常駐服務。

### 6. GTB 的本地模型架構應採雙層或三層，而非單一模型包辦全部

**結論：** GTB 未來應採「快速主路徑模型 + 慢速審查模型」的分層，而不是期待一顆模型同時兼顧即時性與深度判斷。

建議架構：
- 第 1 層：快速主路徑
  - `qwen3:4b` 或 `gemma3n:e2b`
- 第 2 層：備援 / 文字整理 / 長文摘要
  - `gemma3:4b` 或 `qwen3:8b`
- 第 3 層：高風險任務二次審查
  - `deepseek-r1:8b`

這樣的設計可對應 GTB 的真實需求：
- 平常大部分任務走快路徑
- 只有高風險、高不確定度、格式異常或審計需求時，才升級到慢模型複核
