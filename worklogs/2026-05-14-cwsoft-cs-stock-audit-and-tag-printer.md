### 2. 森昱 — iPhone 16 Pro 128G 庫存對不到序號（主案）

客戶反映：庫存查詢表顯示 SHA329312805 有 2 支，但「商品資料」只顯示 1 個序號 `CGYXQPGYCJ-355109182719286`，問哪一天進貨。

#### 對帳結果

- 全部進銷異動明細已整理到 [2026-05-14-cwsoft-senyu-iphone16-phantom-stock.md](2026-05-14-cwsoft-senyu-iphone16-phantom-stock.md)
- 進 4 銷 3 = 算數在庫 **1 支**（`CGYXQPGYCJ-355109182719286`，2025/9/29 進）
- 系統顯示 2 支 = 多 1 支「+1 幽靈」
- 4 筆進貨序號都正常登錄，**不是序號漏填**

#### Root cause

- 「+1 幽靈」最晚 2025-08-14 之前就在了（從「自訂成本」對沖調貨單的數量倒推）
- 落在 2024-12-10 銷貨後、2025-08-14 之間冒出來
- 那筆銷貨單本身正常（未作廢、序號正確），是當時系統沒把庫存扣回去
- 進貨/銷貨明細表都沒有自動寫庫存的 trigger，庫存更新靠 POS 應用程式主動 UPDATE
- 2024-12-26 cwsoft 改過庫存相關 trigger，時間點落在 +1 幽靈出現的區間
- DB 無 audit trail，**從 DB 本身查不出具體哪一步出錯**，需要走 SVN log / POS 程式碼比對才能更深入

#### 客戶端回覆策略

- 「沒序號」是 UI 沒顯示，**不是缺資料**——`庫存查詢表` 畫面只顯示和庫存能對到的序號
- 告訴客戶實體 1 支沒錯，用「盤點 → 盤虧出庫」把系統從 2 改成 1 即可

## 學到的事

### 庫存對帳要用完整公式

只算「進貨 − 銷貨」會跑出一堆假警報。完整公式：

```
算數在庫 = 進貨 + 銷貨退回 + 入庫 + 轉換進 + 出貨退回
        − 銷貨 − 進貨退出 − 出貨 − 出庫 − 轉換出
```

11 種單據都要進來才能跟 `庫存即時用` 對得起來。

### 「自訂成本」操作可以拿來反推歷史 UI 值

`自訂成本` SP 會把當下的 `庫存即時用.店內庫存量` 寫進它建立的兩張對沖調貨單的「數量」欄位。所以這些調貨單就是歷史快照——可以反推某個商品在某個時間點 UI 顯示過多少庫存。這個技巧對追 +1 幽靈很關鍵。

### 客戶端 UI 入口

「庫存查詢表」這個 form（`統計_庫存查詢表_2.vb`）的路徑是 **主選單 → 結帳報表 → 庫存查詢表**，不是即時庫存。點選商品列後右側「商品資料」會顯示該商品在庫的序號清單。

「即時庫存」是用主畫面工具列的「庫存」按鈕（`tb庫存`，工具列第 12 個位置），不在選單裡。
