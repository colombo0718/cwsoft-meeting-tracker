## Root cause 追查

### 確定的事實

1. P241210010007 銷貨單本身正常（分店1、數量1、序號正確、未作廢）
2. 對應的庫存扣減**沒有發生**——從 2025-08-14 自訂成本快照看出當時就多 1
3. 所有交易明細表（進貨物品、銷貨物品、調貨物品）**都沒有寫庫存的 trigger**
4. 庫存更新是 POS 應用程式主動 UPDATE 的，不是 DB 層自動處理
5. 2024-12-26 cwsoft 改了庫存相關 trigger（`庫存修改成本重整` 建立日）——時間點落在 +1 幽靈出現的區間
6. 2025-11-15 起 `庫存修改記錄` 才開始有資料；2025-11-06 `庫存` 表凍結
7. 「+1 幽靈」最晚 2025-08-14 之前就在，且穩定維持

### 系統現在的計算邏輯

```
庫存即時用 = 庫存表（基準值，2025-11-06 凍結）+ 庫存修改記錄.SUM(修改數量)
            = 3 + (-2) = 2  ← 系統顯示
實際對帳   = 進 4 - 銷 3 = 1  ← 應該的數字
```

基準值在凍結時就已經多 1，從此累積錯誤無法自癒。

### 無法從 DB 溯源的部分

- 沒有「每天庫存即時用是多少」的歷史快照
- 沒有 2025-11-15 之前的庫存異動 audit log
- 任何直接 UPDATE 庫存即時用 / 庫存表 的人工操作都沒留痕跡
- POS 應用程式當時的程式碼版本和執行 log 不在 DB 內

### 不是這些原因

- **不是當前 trigger 邏輯 bug**：trigger 程式碼本身正確，問題在基準值
- **不是商品複製沒填序號**：4 筆進貨的序號都正常登錄
- **不是商品合併/轉換**：商品 2024-11-23 才建立，沒有商品編號修改痕跡，無公版編號

### 接下來只能往 DB 外查

1. POS 應用程式的 SVN log（2024-12 期間銷貨/庫存相關 commit）
2. POS 應用程式的 log（員工 1099 在 2024-12-10 18:53:23 那筆銷貨當下程式有沒有報錯）
3. `PosConfig.dbo.錯誤清單` 表，可能有 POS 程式 catch 過的錯誤記錄
